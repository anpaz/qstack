\section{Examples}\label{sec:examples}

A qstack program is a \emph{kernel} of the form:

\begin{minted}{ruby}
allocate q:
  instructions
measure callback
\end{minted}

where \texttt{allocate} and \texttt{measure} are keywords, \texttt{q} is the name of a new qubit, initially $0$ (technically $\ket{0}$), used in \texttt{instructions}, and then measured and passed to the classical code \texttt{callback}. Each instruction is either a quantum gate or another kernel.

Our implementation in Python (see Section~\ref{sec:implementation}) is an embedded Domain Specific Language (DSL), where quantum programs are represented using Python data structures. The \texttt{Kernel} class encapsulates a qubit name, a list of instructions, and a callback name. Callbacks are Python functions that can dynamically generate additional quantum computation by returning new \texttt{Kernel} objects. Throughout our examples, for improved readability, callbacks typically create string representations of kernels and use \texttt{parse\_kernel} to convert them to the proper data structure, but strings and parsing are not necessary as we have the full power of an embedded DSL.

\subsection{Hello \textit{entangled} world!}

Consider a simple quantum program that creates the so-called Bell state:

\begin{minted}{ruby}
allocate q1:
  mix q1
  allocate q2:
    entangle q1 q2
  measure collect
measure collect
\end{minted}

This program illustrates several elements of qstack:

\textbf{Quantum instructions (gates):} Instructions like \texttt{mix} and \texttt{entangle} are quantum operations that manipulate quantum states. These belong to qstack's ``toy instruction set'' (a collection of intuitively named gates useful for introductory examples. The \texttt{mix} operation creates a superposition state where a qubit exists as both $0$ and $1$ with equal probaibility. The \texttt{entangle} operation creates quantum correlation between qubits; measuring one qubit instantly determines the state of the other, producing perfectly correlated outcomes. (We explore how these operations map to standard quantum gates in Section~\ref{sec:synthesis}.)

\textbf{Nested kernels:} A kernel can appear anywhere an instruction is expected, enabling dynamic program structure. In our example, the inner kernel \texttt{allocate q2: entangle q1 q2; measure collect} is itself an instruction within the outer kernel. 

\textbf{Measurement and callbacks:} When a kernel completes its instructions, it measures its allocated qubit, which has several effects: the qubit is deallocated from the quantum system, any entanglement with other qubits causes those qubits to potentially collapse into a subset of their possible states, and the classical measurement result ($0$ or $1$) is passed to the specified callback function. In our Bell state example, both measurements use the \texttt{collect} callback, which simply stores each result on a classical stack without generating additional quantum computation.

When this Bell state program runs in a noiseless environment, the expected outcome is perfectly correlated measurement results: both qubits will measure as $0$ about $50\%$ of the time, and both will measure as $1$ about $50\%$ of the time. Unless there is noise, the result will never be $(0,1)$ nor $(1,0)$.

\subsection{Majority vote}\label{sec:majority-vote}

Critical for quantum error correction is the ability to compile quantum programs into versions that encode a single logical qubit using multiple physical qubits. Consider this simple program that returns $1$ by applying a \texttt{flip} operation to a qubit:

\begin{minted}{ruby}
allocate q:
  flip q
measure collect
\end{minted}

Without noise, this program always returns $1$. However, in real quantum hardware with noise, if the error rate for a flip operation is $p$, the program incorrectly returns $0$ with probability $p$.

To increase reliability, a simple repetition code can encode a single logical qubit using three physical qubits, reducing the error probability from $p$ to approximately $3p^2(1-p) + p^3$ when $p < 0.5$. With $p = 0.1$, the error rate drops from 10\% to approximately 2.8\%. Note this repetition code is \textit{not} a complete quantum error correction technique, as it only addresses bit-flip errors but not phase-flip errors. We use it purely for pedagogical purposes, while proper quantum error correction codes, like the Steane code discussed in section~\ref{sec:error-correction}, handle both types of errors.

\begin{minted}{ruby}
allocate q_1: allocate q_2: allocate q_3:
  flip q_1
  flip q_2
  flip q_3
measure collect measure collect measure vote
\end{minted}

The \texttt{vote} callback implements the majority voting process:

\begin{minted}{python}
def vote(m1):
    m2 = context.pop()
    m3 = context.pop()
    if (m1 + m2 + m3) >= 2:
        context.push(1)  # Majority voted for 1
    else:
        context.push(0)  # Majority voted for 0
    return None
\end{minted}

This callback consumes the three individual measurement outcomes, computes their majority vote, and places a single logical result back onto the stack. Since \texttt{vote} returns \texttt{None}, there is no further quantum computation.  The error correction happens entirely within the classical processing while preserving the program's observable behavior: a single measurement outcome that semantically matches the original program, so any enclosing computation need not know that qubits and classical computations were inserted.

The measurement stack is a purely classical idiom, but used commonly enough that \texttt{collect}, which pushes the measurement onto the stack, is the default behavior. We also allow syntactic sugar for allocating multiple qubits. When using this syntactic sugar, if a callback is specified, it automatically receives multiple measurement outcomes, one for each qubit being measured, rather than just a single measurement. This sugar is easily implemented by the compiler wrapping the program's multiple-argument callback with a callback that does the straightforward stack management. Using this feature, the example above could be written more concisely as:
\begin{minted}{ruby}
allocate q_1 q_2 q_3:
  flip q_1
  flip q_2
  flip q_3
measure vote
\end{minted}

\begin{minted}{python}
def vote(m1,m2,m3):
    if (m1 + m2 + m3) >= 2:
        context.push(1)  # Majority voted for 1
    else:
        context.push(0)  # Majority voted for 0
    return None
\end{minted}
As described later, we expect error correction transformations like this repetition code to be implemented as compiler passes that can be applied to any quantum program. This separates higher-level algorithm implementation from error-correction implementation.

\subsection{Creating Magic}

The next example demonstrates qstack's ability to implement advanced quantum operations through classical feedback. The T-gate, a crucial operation for universal quantum computation, applies a $\pi/4$ phase rotation that takes a qubit outside the ``Clifford group'' of operations. While Clifford operations alone can be efficiently simulated classically, adding T-gates enables universal quantum computing with the computational power to solve problems believed to be intractable on classical computers. In fault-tolerant quantum computers, T-gates are typically implemented indirectly through ``magic state injection''~\cite{bravyi2005universal, Litinski2019gameofsurfacecodes}, a technique that uses Clifford gates plus specially prepared resource states. This approach is necessary because direct T-gate implementation is challenging on many hardware platforms and many error correction codes. The procedure entails creating a prepared ``magic state'', entangling it with the target qubit, performing a measurement, and then conditionally applying a Clifford correction based on the measurement outcome. %Magic states can be further refined through distillation protocols that improve their quality through multiple rounds of purification. 
 In qstack, this technique looks like:

\begin{minted}{ruby}
allocate target:
  mix target          
  allocate resource:
    prepare_magic_state resource      
    entangle target resource      
  measure t_correction    
  mix target
measure collect                   
\end{minted}

The \texttt{t\_correction} callback conditionally applies a correction based on measurement results:

\begin{minted}{python}
def t_correction(measurement):
    if measurement == 1:
        return parse_kernel("""
          allocate _:
            s target_qubit
          measure discard
        """)
    else:
        return None
\end{minted}

In this implementation, the \texttt{t\_correction} callback encapsulates the logic needed for the final correction step. Unlike the \texttt{repeat\_until\_zero} example in the introduction, this correction is not iterative: it applies a deterministic fix-up operation based on the measurement outcome and is fundamental to the algorithm itself. This conditional logic pattern is common in quantum algorithms, and while most languages support it through special-purpose classical control constructs (like \texttt{if\_test} in Qiskit), qstack handles it using the same callback mechanism as all other classical processing, maintaining a consistent programming model.

While the kernel returned by the callback allocates an unused qubit (denoted by \texttt{\_}) solely to provide a scope for applying the S-gate operation on the target qubit (and \texttt{discard} ignores its argument and returns \texttt{None}), we expect an actual compiler to avoid actually allocating such a qubit.

The entire procedure correctly implements a T-gate. When evaluated, the target qubit's final probability distribution will be approximately $\cos^2(\pi/8) \approx 85\%$ probability of measuring $0$ and $\sin^2(\pi/8) \approx 15\%$ probability of measuring $1$.

\subsection{Compositional Compilation}

A key advantage of qstack's design is the ability to chain multiple compilation stages, where each stage transforms the program to target different abstraction levels. To demonstrate this powerful capability, let's consider two sequential compilation examples:

\begin{minted}{python}
toy_program = parse_kernel("""
allocate q:
  mix q
measure my_callback
""")

canonical_compiler = ToyCompiler()
canonical_program = canonical_compiler.compile(toy_program)

rep3_compiler = TrivialRepetitionCompiler()
error_corrected_program = rep3_compiler.compile(canonical_program)
\end{minted}

On a first round, the program is compiled into one that uses canonical quantum instructions. On a second round, basic error correction is introduced using a repetition code, which involves adding both quantum and classical computation. Compilation is typically modular and compositional --- this example would have worked just as well in the opposite order.

As Section~\ref{sec:callbacks} explains, we also need the compilation to occur later on kernels dynamically generated by callbacks, which must occur during program execution once the generated kernels are known. This is essentially like \verb|eval| in many programming languages and is entirely classical work to create quantum instructions.

This chaining capability is powerful. Each compiler performs its specialized transformation: converting between gate sets, adding error correction, optimizing resource usage, etc. %while ensuring that the callbacks continue to function correctly.
